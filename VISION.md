# CodeCanvas Vision

## 한 줄 정의

CodeCanvas는 코드를 읽지 않아도 로직 흐름을 이해하고 리뷰할 수 있게 만드는 시각화 도구다.

## 우리가 만들고 싶은 것

우리가 만들고 싶은 것은 단순한 코드 맵이 아니다.

- 축소하면: 요청 또는 실행 단위의 로직 설명기
- 확대하면: 의존성, 파일, 함수, 분기 구조를 보여주는 코드 구조 탐색기

즉 CodeCanvas는 `코드 시각화 툴`이면서 동시에 `코드 리뷰 설명기`여야 한다.

## 핵심 사용자 약속

사용자는 다음 질문에 코드 원문을 길게 읽지 않고 답을 얻어야 한다.

- 이 요청은 어디로 들어오는가
- 어떤 인증/의존성/서비스/DB/API를 거치는가
- 어떤 분기 때문에 이 응답이 나오는가
- 어디가 위험한가
- 어디를 리뷰해야 하는가

## 제품 관점의 목표

### 1. 실행 흐름을 먼저 보여준다

기본 화면은 전체 코드 구조가 아니라 실제 엔트리포인트 기준 흐름이어야 한다.

- Request
- Route
- Dependency
- Service
- Repository / DB
- External API
- Response

### 2. 확대할수록 코드 구조를 드러낸다

줌인할수록 사용자는 더 구체적인 구현을 볼 수 있어야 한다.

- Level 0: 요청 설명
- Level 1: 레이어 설명
- Level 2: 파일 / 클래스 설명
- Level 3: 함수 / 메서드 설명
- Level 4: 분기 / 예외 / 쿼리 / 도구 호출 설명

### 3. 리뷰 관점 정보를 같이 보여준다

단순히 "어디를 탔는가"만 보여주면 부족하다.
리뷰에 필요한 정보가 함께 보여야 한다.

- 인증 경로
- 예외 경로
- 데이터 변경 지점
- 외부 API 호출 지점
- 불확실한 연결
- 실제 실행된 경로와 가능한 경로의 차이

## 우리가 추가하는 것

기존의 일반적인 코드 시각화 도구와 달리 CodeCanvas는 아래를 추가한다.

### 1. Semantic Zoom

줌아웃은 요약이 아니라 실행 설명이어야 하고, 줌인은 나열이 아니라 구조 확장이어야 한다.

- Zoom out = explain
- Zoom in = inspect

### 2. Runtime-first View

정적 분석은 보조 정보이고, 기본 뷰는 실제로 탄 경로를 중심으로 보여줘야 한다.

- 실제 실행 경로
- 실제 응답
- 실제 예외
- 실제 DB/API 호출

### 3. Review Overlay

그래프 위에 리뷰에 필요한 신호를 올린다.

- 위험 노드
- 인증 / 권한 체크 누락 후보
- 실패 가능 분기
- 외부 의존성 집중 구간
- 변경 영향 범위

### 4. Evidence and Confidence

모든 노드와 엣지는 확신 수준과 근거를 가져야 한다.

- definite
- inferred
- runtime

사용자는 "확정 사실"과 "추정"을 구분할 수 있어야 한다.

## 이 도구가 해결하려는 문제

AI로 인해 코드 생산량은 계속 증가한다.
문제는 코드를 만드는 속도가 아니라 코드를 이해하고 리뷰하는 속도다.

지금 필요한 것은 더 많은 코드 생성기가 아니라 아래를 가능하게 하는 인터페이스다.

- 코드베이스를 빠르게 이해하는 것
- 요청 단위로 로직을 검증하는 것
- 위험 지점을 빠르게 찾는 것
- 코드를 다 읽기 전에 리뷰 판단을 내리는 것

## 제품 원칙

### 1. 그래프는 사실을 숨기지 않는다

- 실제 경로와 가능한 경로를 섞지 않는다
- 추론 결과를 확정처럼 보여주지 않는다
- 불확실하면 불확실하다고 표시한다

### 2. 구조보다 설명이 우선이다

그래프의 목적은 복잡한 연결을 많이 보여주는 것이 아니라
"왜 이 결과가 나왔는지"를 설명하는 것이다.

### 3. 코드를 안 읽어도 1차 리뷰가 가능해야 한다

사용자는 그래프만 보고도 다음 판단을 할 수 있어야 한다.

- 인증이 맞는가
- 분기 처리가 안전한가
- DB/API 호출이 적절한가
- 실패 경로가 있는가
- 더 깊게 들어가야 할 지점이 어디인가

## 이상적인 사용자 경험

1. 저장소를 연다
2. 엔트리포인트를 고른다
3. 실제 요청을 넣거나 샘플 요청을 실행한다
4. Level 0에서 전체 흐름과 응답 이유를 본다
5. 이상한 노드를 더블클릭해 Level 2~4로 들어간다
6. 코드 원문은 정말 필요할 때만 연다

## 성공 기준

이 도구가 성공했다는 것은 사용자가 다음 말을 할 수 있는 상태다.

- 코드를 다 읽지 않았지만 흐름은 이해했다
- 어디를 리뷰해야 하는지 바로 알겠다
- 왜 이 응답이 나왔는지 설명할 수 있다
- 위험 지점을 먼저 보고 필요할 때만 코드를 열면 된다

## MVP 기준 해석

MVP의 목표는 "예쁜 그래프"가 아니다.

MVP의 목표는 아래 하나다.

`한 개의 요청 흐름을 코드 원문 없이 설명하고 리뷰 포인트를 찾게 만드는 것`

## 이 문서와 SPEC의 관계

- `SPEC.md`: 구현 범위와 기술 명세
- `VISION.md`: 제품이 무엇이어야 하는지에 대한 기준

구현 우선순위가 헷갈릴 때는 항상 이 문서 기준으로 판단한다.
